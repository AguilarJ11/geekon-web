Postulaciones

El perfil contiene una sección de postulaciones.

Las postulaciones no utilizan categorías hardcodeadas.

El administrador crea formularios desde:

/admin/formularios

Cuando un formulario está publicado, puede aparecer como una postulación disponible para el usuario.

El perfil debe mostrar:

Formularios publicados disponibles
Formularios que el usuario todavía no respondió
Formularios ya enviados
Estado de cada postulación

Estados posibles:

Pendiente
Aprobada
Rechazada

Nunca crear lógica específica basada en nombres como:

Artista
Stand Comercial
Concurso de Cosplay
Torneo de TCG
Torneo de Videojuegos

Todos deben funcionar mediante el mismo sistema de formularios.

Roles

IMPLEMENTADO (2026-07-23). El perfil muestra una fila de badges de rol,
no uno solo — un usuario puede tener varios a la vez:

1. Rol de staff (si `user.role` es ADMIN/STAFF_CM/STAFF_STANDS/COORDINATOR).
2. Un rol de "participante" por cada categoría de formulario en la que
   tenga al menos una postulación APROBADA (ej. Cosplayer, Artista,
   Competidor, Comerciante, Asistente).
3. Un rol de "organizador" por cada formulario del que sea dueño
   (ownerId), derivado de la categoría de ese formulario (ej.
   "Organizador de Torneo", "Expositor / Taller").

Si no tiene ninguno de los anteriores, se muestra "Visitante".

La fuente de datos explícita es el campo `Form.category` (enum fijo:
ARTE, STAND, COSPLAY, TORNEO, ACTIVIDAD, OTRO), elegido por el admin al
crear el formulario — nunca se infiere del título/nombre del
formulario. Ver `src/lib/form-categories.ts` (catálogo con
participantRole y ownerRole por categoría) y `src/app/perfil/page.tsx`
(cálculo de `roles`).

Dueño de formulario: el admin puede asignarle a cualquier usuario (por
email) la propiedad de un formulario puntual desde
/admin/formularios/[id]/editar. Ese dueño administra su propio
formulario desde /mis-formularios (fuera del panel admin), sin acceso a
otros formularios. Ver `requireFormAccess()` en `src/lib/auth.ts`.

Reglas de implementación

Cada funcionalidad debe implementarse de manera individual.

Antes de modificar el perfil:

Revisar los componentes existentes.
Revisar el modelo de usuario.
Revisar las APIs y acciones existentes.
Revisar si existe una solución reutilizable.
Evitar duplicar componentes o esquemas.

Nunca:

Rehacer toda la página por una funcionalidad pequeña.
Cambiar estilos que no estén relacionados con la tarea.
Eliminar campos existentes sin una instrucción explícita.
Hardcodear tipos de postulaciones.
Crear roles basados únicamente en nombres de formularios.
Implementar varias fases del roadmap en un mismo cambio.
Introducir datos de ejemplo en producción.

Siempre:

Mantener compatibilidad con los perfiles existentes.
Validar los datos tanto en frontend como en backend.
Mantener el diseño responsive.
Contemplar estados de carga, error y vacío.
Probar que el usuario solo pueda editar su propio perfil.
Documentar las decisiones nuevas.
Actualizar el roadmap al completar una funcionalidad.
Sistema de roles según participación — RESUELTO (2026-07-23)

Qué roles existen inicialmente: ver catálogo en src/lib/form-categories.ts
(uno participante y uno organizador por categoría) + roles de staff
existentes (ADMIN/STAFF_CM/STAFF_STANDS/COORDINATOR).

Un usuario puede tener varios roles: sí, se muestran todos como fila de
badges en /perfil.

Quién asigna los roles: se derivan automáticamente — el rol de
participante sale de tener una postulación APROBADA en esa categoría; el
rol de organizador sale de ser dueño (ownerId) de un formulario de esa
categoría. Nadie los asigna a mano.

Si afectan permisos o son solo visuales: mixto. El rol de organizador SÍ
afecta permisos reales (acceso a /mis-formularios/[id] para ese
formulario puntual, vía requireFormAccess()). Los demás son solo
visuales en el perfil.

Relación con postulaciones aprobadas: directa — ver "Roles" más arriba.

Próxima funcionalidad: Fase 4 del roadmap (seguir/agregar amigos,
portfolio, privacidad, historial de ediciones).
